Mes — Hermes Agent Planning Document

Planning & reference document

Purpose of this doc: A reference and direction-setter for building "Mes," a personal Hermes Agent profile/distribution. This is a guiding document, not a rigid spec. An implementing agent should treat it as intent + guardrails, and is free to make better implementation choices where they serve the goals below.

Implementation detail: This document should stay mid-to-high level. Lower-level commands, candidate packages, schemas, and templates live in Mes-Implementation-Notes.md and must be re-checked against official docs before use.

1. Vision

Mes is a named Hermes Agent profile/distribution that helps with day-to-day work tasks. It acts as the user (delegated identity), runs locally through Hermes-supported local/container execution, and orchestrates sub-agents and tools to read, summarize, research, and (eventually) take action across the user's work ecosystem.

For now, Mes should be a terminal-first tool. The CLI is the main entry point for chat, one-off commands, ingestion jobs, summaries, and admin/config tasks. A web UI can come later if the CLI workflows prove useful.

Guiding principles:

  • Least privilege first. Read-only before write. Scoped credentials per capability.
  • Acts as the user. Delegated permissions — Mes can only see/do what the user can.
  • Local & private. Runs on the user's local machine; no inbound exposure.
  • CLI-first. Optimize the early product for fast terminal workflows before building a UI.
  • Composable. Capabilities added in phases; sub-agents own narrow domains.
  • Hermes-native first. Use profile/distribution, config, MCP, sessions, skills, delegation, logs, and model routing built into Hermes before adding custom Mes code.
  • Not over-constrained. Implementation details are suggestions; optimize for the goals.

3. Capabilities (Target State)

Read / Ingest

  • ✓Read user's emails or run email intake automation only if Microsoft Graph delegated mail access becomes available; otherwise treat email read/intake as deferred.
  • ✓Read Teams service data only if Graph access is approved. This includes chats, channel messages, meetings, transcripts, and related meeting artifacts; transcripts also require retention and user-visible rights.
  • Ingest manually exported emails, meeting notes, or transcript files into Obsidian/Markdown as the near-term fallback path.
  • Read and maintain Obsidian notes from a local vault using a Markdown-first system; optionally add RAG later.
  • Read Azure DevOps work items, wiki, repos, pull requests, builds, pipelines, iterations, and test plans through an approved MCP path where practical.
  • Read Azure cloud resource inventory, storage metadata, app configuration, logs, and diagnostics through Azure MCP in read-only mode with approved service scope.
  • Read GitHub wiki and repos
  • Query Entra for people and team context through the user's existing approved access path.
  • Search public web/news sources for recurring briefing topics, with source links and date filters. Do not leak internal/private work terms into public search unless the user explicitly approves that scope.

Reason / Produce

  • Summarize emails, meetings, notes
  • Research mode — strong internet research
  • Model routing via LiteLLM — pick the right proxy model alias per task (cheap/fast for extraction & classification; strong for summarization & planning)
  • Sub-agents — the main agent can create bounded-scope sub-agents for specific tasks, choose their model route, and delegate only the MCP access needed for the task. V1 should not allow recursive delegation.
  • Planning & proposals — make plans, propose actions, and prepare coding handoff documents for separate coding agents or humans
  • Weekly briefing — produce a concise, dated research artifact covering user-selected topics such as work-adjacent updates, new AI models, software engineering news, and relevant industry movement.
  • Todo list with notes — keep human-editable todos and follow-ups in Obsidian, with links back to notes and source artifacts. Mes must tolerate the user directly editing these todos.
  • Agent work queue — add Hermes Kanban when tasks need durable assignment, retries, worker profiles, cross-agent handoff, or long-running coordination.
  • Answer composite queries, e.g. "What was X meeting about, and is there anything I need to do?" using the Markdown knowledge base, transcripts, and todo state; add RAG only if direct file/wiki lookup is insufficient

Act (Later Phases)

  • Draft emails
  • Draft Azure DevOps work items with proposed fields, links, sprint/iteration placement, and rationale; create/update only after user approval.
  • Draft GitHub issues, PR comments/reviews, and coding handoff documents before posting or acting. Mes and its sub-agents should not edit code directly.
  • Do not mutate Azure cloud resources. Azure MCP is for read-only inventory, logs, service details, diagnostics, and command/runbook suggestions.
  • Intake email ingestion and extraction only after Graph/mail access is feasible.

5. Security & Auth Posture

Mes should stay local, delegated, and least-privilege. The first useful version should work without broad Microsoft Graph access and without any Azure cloud mutation capability.

  • Delegated identity: Mes acts as the user and should only see or do what the user can already access.
  • Read-only default: External systems start read-only. ADO and GitHub writes are proposal-first; Azure cloud remains read-only.
  • Runtime secrets: Keep tokens and provider credentials out of the profile repo and image. Prefer runtime-mounted secrets and scoped credentials.
  • Graph dependency: Microsoft 365 email, Teams service data, and transcripts are deferred until delegated Graph access is realistically available.
  • State boundary: Human knowledge belongs in Obsidian. Operational state belongs in Hermes/Mes runtime state.

6. Operating Model

Mes should begin as a Hermes profile distribution, not a separate application. Use Hermes-native profile, MCP, skill, cron, memory, session, delegation, and model-routing behavior before adding Mes-specific wrappers.

  • Primary surface: Terminal-first through the Mes Hermes profile, with a thin alias only if it improves daily use.
  • Model gateway: Route through LiteLLM aliases so providers and concrete model names can change outside Mes.
  • Sub-agents: The main agent may create bounded-scope sub-agents for specific tasks, choose their model route, and delegate only the MCPs needed for the task.
  • Recurring work: Start recurring research and summaries manually; schedule through Hermes cron only after the output is useful and explicitly approved.
  • Inspection: Prefer Hermes-native logs, sessions, MCP status, and agent surfaces before inventing Mes-specific operational tooling.

8. Key Workflows

  • ADO drafting: Use live ADO data plus human-maintained board context to propose work item changes. User approval is required before any write.
  • Coding handoffs: Mes and sub-agents prepare handoff documents for humans or separate coding tools; they do not edit code directly.
  • Weekly briefings: Produce curated, dated, source-linked briefings on user-selected topics such as work-adjacent developments, AI models, software engineering, and industry news.
  • Obsidian knowledge: Keep Obsidian as human-readable knowledge, todos, summaries, and decisions. Avoid turning it into a hidden operational database.
  • Kanban later: Use Hermes Kanban only when work needs durable agent assignment, retry/block state, comments, or multi-run coordination.

9. V1 Boundary

V1 should prove Mes as a useful Hermes profile distribution. It should launch locally, load approved MCPs, answer questions over local and developer-system context, draft external work without directly mutating it, and delegate bounded sub-agent work with parent-selected model routes.

Included in V1Excluded from V1
Hermes profile/distribution assets, Obsidian shared knowledge access, ADO/GitHub/Azure read context, LiteLLM model aliases, bounded sub-agent delegation, proposal-first external writes, coding handoffs, and manual weekly briefing workflow. Microsoft Graph email/Teams dependency, automatic email intake, RAG/vector indexing, Hermes Kanban, recursive agents, direct Azure resource mutation, Mes-authored code changes, and broad unattended note editing.

V1 done criteria: Mes can launch as a Hermes profile, load approved MCPs, query Obsidian plus Azure DevOps, GitHub, and Azure cloud read-only context, produce an Obsidian handoff or output artifact, and delegate one bounded sub-agent with a selected model route and restricted delegated MCP set.

10. Phased Direction

  1. Local read-only foundation: Establish the Hermes profile distribution, model routing, Obsidian read access, manual artifact ingestion, and basic summarization.
  2. Developer-system context: Add approved read paths for Azure DevOps, GitHub, Azure cloud, and Entra context.
  3. Planning outputs: Make ADO drafts, GitHub drafts, coding handoffs, and weekly briefings useful before enabling any recurring or write behavior.
  4. Coordination expansion: Add Hermes Kanban, RAG, Graph-backed sources, and more automation only after local workflows prove their value.

11. Open Decisions

  • Profile shape: Finalize the first native Hermes profile distribution repo contents.
  • MCP policy: Decide whether Hermes-native MCP config is enough or whether Mes needs a thin policy layer around approved servers and delegated MCP sets.
  • Obsidian scope: Choose the vault path, readable folders, and constrained write locations.
  • ADO context: Define where stable board conventions live and how dynamic sprint context is maintained.
  • Weekly briefing: Define the editable topic set, source preferences, privacy exclusions, and when the workflow is ready for scheduling.
  • Graph feasibility: Confirm whether delegated Graph access is realistic before planning email or Teams service integrations.
This document is intended as direction and context. An implementing agent should follow the intent and guardrails (least privilege, delegated identity, local hardened container, phased rollout) while remaining free to choose better implementation details where warranted.
